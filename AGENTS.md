# Repository Guidelines

## Project Structure & Module Organization

- `Cargo.toml` defines the Rust package, features, and binary target (`dbx`); update dependency versions and crate metadata there.
- `build.rs` generates the Cap'n Proto artifacts used by the CLI and server layers, so keep schema-related build changes aligned with it.
- `src/main.rs` is the CLI entry point. Command handlers live under `src/commands/`, shared runtime and storage logic live under `src/lib/`, and logging setup lives in `src/logging.rs`.
- Add reusable logic to `src/lib/` and keep `src/commands/` focused on argument parsing, orchestration, and user-facing output.
- Integration and regression coverage live in `tests/` (`cli_smoke.rs`, `cli_help.rs`, `control_regression.rs`, `plugin_regression.rs`). Keep unit tests next to the code they exercise with `#[cfg(test)]`.
- `README.md` is the product and CLI reference. `package.json` and `bin/` support the npm wrapper for the `dbx` binary; update them when install or packaging behavior changes.
- `target/` is generated by Cargo and should stay untracked.

## Build, Test, and Development Commands

- `cargo check` for a fast compile-only pass while iterating on Rust changes.
- `cargo fmt -- --check` to verify formatting before review, or `cargo fmt` to apply rustfmt.
- `cargo clippy --all-targets --all-features` to catch lint regressions before opening a PR.
- `cargo test` is the baseline verification suite and must be run before every commit.
- `cargo test --test cli_smoke -- --nocapture` or `cargo test --test plugin_regression -- --nocapture` to focus on a single integration suite while debugging.
- `cargo run -- <args>` to exercise the CLI locally, for example `cargo run -- start --foreground`.
- `cargo build --release` when validating optimized binaries or packaging behavior.

## Contributor Workflow

- Start by checking the worktree state with `git status --short` so you do not overwrite unrelated local changes.
- Keep changes scoped. Prefer small patches that touch either a command module, a library module, or a single test area when possible.
- Run `cargo fmt -- --check`, `cargo clippy --all-targets --all-features`, and `cargo test` before opening a PR.
- At minimum, run `cargo test` before creating a commit. If you cannot run the full suite, note the gap in your handoff or PR.
- When changing CLI behavior, update `README.md` and add or adjust a regression test under `tests/`.
- When changing shared storage, schema, plugin, or replication logic, prefer adding targeted unit coverage in `src/lib/` plus an integration test when the behavior crosses process boundaries.

## Coding Style & Naming Conventions

- Follow rustfmt defaults: 4-space indentation, trailing commas, and grouped imports when rustfmt applies them.
- Use snake_case for functions, modules, and variables; CamelCase for types and enums; SCREAMING_SNAKE_CASE for constants.
- Keep command modules thin and move reusable logic into `src/lib/` instead of duplicating behavior across CLI handlers.
- Add `///` comments for externally visible APIs and short inline comments only where intent is not obvious from the code.
- Prefer explicit error propagation with context over `unwrap`/`expect` in production paths.

## Testing Guidelines

- Co-locate unit tests with implementation and give tests descriptive names such as `rejects_invalid_token_scope`.
- Mirror integration test filenames to the behavior under test and extend the existing regression suites instead of creating overlapping coverage.
- Cover happy paths and failure modes, especially CLI parsing, schema enforcement, plugin dispatch, replication, and control-plane behavior.
- Re-run `cargo test` after modifying tests or shared runtime code. Use targeted suites during iteration, but finish with the full test run before commit.

## Commit & Pull Request Guidelines

- Use Conventional Commits for every commit, such as `feat: add schema publish audit trail`, `fix: preserve plugin retry delay`, or `docs: refresh contributor workflow`.
- Keep commits focused and do not mix unrelated refactors, dependency updates, and feature work in the same commit.
- Include a commit body when the rationale or migration impact is not obvious from the subject line.
- Reference issue IDs when available, for example `fix: guard remote sync retry (#42)`.
- PRs should summarize the user-visible impact, list the verification commands you ran, and call out any follow-up work or known gaps.
